# 飲食店向け 備品発注管理システム

### **【開発意図】3ヶ月でのリリース完遂と、テスト工程の徹底による品質担保**
元店長のドメイン知識をベースに、3ヶ月という限られた工期で実用的な機能を厳選し、Java/Spring Bootで実装しました。テスト工程の実務経験を開発にフィードバックし、手戻りの少ない設計と検証を徹底しています。

---

## 1. プロジェクト概要
現場の「発注ミス」と「管理コスト」を削減するためのWebアプリケーションです。
短期間開発ながら、「ユーザーが迷わない操作性」と「開発現場での保守性」を両立させています。

* **Demo URL**: [https://restaurant-supply-order.onrender.com]
* **テスト用アカウント（店舗権限）**: `tenpo01` / `tenpo`
* **テスト用アカウント（本部権限）**: `honbu_user` / `honbu`
    * ※[デモ用アカウント一覧](#5-デモ用ログインアカウント一覧)に他店舗のIDも記載しています。

---

## 2. 現場視点で厳選した「優先実装機能」
3ヶ月という工期（納期）を遵守するため、ユーザビリティと実務上の重要度から以下の3点をコア機能として選定・実装しました。

1.  **ステータス履歴の可視化**:
    * 「発注中」「承認済み」「完了」といった履歴を追跡可能にし、二重発注や確認漏れを防ぎます。
2.  **パスワード変更機能**:
    * 各店舗でのセキュリティ意識向上とアカウント管理を想定し、ログイン後の自己管理機能を実装。
3.  **ログインユーザーの動的表示**:
    * 複数店舗の並行操作ミスを防ぐため、常に「どの店舗（〇〇店店長）」として操作しているかをヘッダーに可視化。

---

## 3. 品質管理（テスト工程の実施）
テスト現場での実務経験を活かし、開発工程の後半をテストに充てることで、信頼性の高いシステムを構築しました。

* **テスト設計**: 境界値分析や同値分割法を用い、異常系（不正な値の入力や未ログイン時のアクセス制限）のテストケースを作成。
* **実施内容**:
    * **単体テスト**: JUnitによるビジネスロジックの検証。
    * **結合テスト**: 画面遷移、DB連動、セッション維持の正常性確認。

* **アジャイルな開発意識**: 「まずは動くものを3ヶ月で」という目標を立て、優先順位を明確にして開発を完遂しました。
